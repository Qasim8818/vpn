01 P4 / SmartNIC — Programming the Network in Silicon HARDWARE

XDP runs on the host CPU. DPDK dedicates host CPU cores to polling. P4 goes one level deeper: you program the NIC silicon itself. The packet processing logic runs on the NIC's embedded processor or ASIC, consuming zero host CPU cycles. At 100 Gbps+, this is the only realistic path.

1.1 — P4 Language: Programming the Data Plane

P4 (Programming Protocol-independent Packet Processors) is a DSL that compiles to NIC hardware targets — Tofino switches, Netronome Agilio, Nvidia (Mellanox) BlueField DPUs. It describes exactly how packets are parsed, matched, and acted upon in silicon.

// P4 program: WireGuard-aware NIC — offload decryption decision to hardware
// Target: Netronome Agilio / Nvidia BlueField-3 DPU

P4_16
header ethernet_t {
    bit<48> dst_addr;
    bit<48> src_addr;
    bit<16> ether_type;
}

header ipv4_t {
    bit<4>  version;
    bit<4>  ihl;
    bit<8>  diffserv;
    bit<16> total_len;
    bit<16> identification;
    bit<3>  flags;
    bit<13> frag_offset;
    bit<8>  ttl;
    bit<8>  protocol;
    bit<16> checksum;
    bit<32> src_addr;
    bit<32> dst_addr;
}

header udp_t {
    bit<16> src_port;
    bit<16> dst_port;
    bit<16> length;
    bit<16> checksum;
}

// WireGuard transport message header
header wg_transport_t {
    bit<8>  msg_type;       // 0x04 = data
    bit<24> reserved;
    bit<32> receiver_index;
    bit<64> counter;        // Nonce — monotonic per session
}

struct metadata_t {
    bit<1>  is_wireguard;
    bit<1>  replay_detected;
    bit<32> peer_id;
}

// Parser: stateful packet parsing in hardware
parser IngressParser(
    packet_in pkt,
    out headers_t hdr,
    out metadata_t meta,
    out ingress_intrinsic_metadata_t ig_intr_md)
{
    state start {
        pkt.extract(hdr.ethernet);
        transition select(hdr.ethernet.ether_type) {
            0x0800: parse_ipv4;
            default: accept;
        }
    }
    state parse_ipv4 {
        pkt.extract(hdr.ipv4);
        transition select(hdr.ipv4.protocol) {
            17: parse_udp;     // UDP
            default: accept;
        }
    }
    state parse_udp {
        pkt.extract(hdr.udp);
        transition select(hdr.udp.dst_port) {
            51820: parse_wireguard;    // WireGuard port
            default: accept;
        }
    }
    state parse_wireguard {
        pkt.extract(hdr.wg_transport);
        meta.is_wireguard = 1;
        transition accept;
    }
}

// Match-action tables: O(1) lookup in TCAM/SRAM
control IngressControl(
    inout headers_t hdr,
    inout metadata_t meta,
    in ingress_intrinsic_metadata_t ig_intr_md,
    inout ingress_intrinsic_metadata_for_tm_t ig_tm_md)
{
    // Counter table: last seen counter per receiver_index
    // Stored in NIC SRAM — no host memory access
    Register<bit<64>, bit<32>>(65536) last_counter;

    action drop_packet() {
        ig_tm_md.drop_ctl = 1;
    }
    action send_to_host() {
        ig_tm_md.ucast_egress_port = CPU_PORT;
    }

    // Replay check — executes entirely in NIC at 100 Gbps
    action check_replay() {
        bit<64> last_val;
        last_counter.read(last_val, hdr.wg_transport.receiver_index);
        if (hdr.wg_transport.counter + (bit<64>)0x80000000 < last_val) {
            meta.replay_detected = 1;
        } else if (hdr.wg_transport.counter > last_val) {
            last_counter.write(hdr.wg_transport.receiver_index,
                               hdr.wg_transport.counter);
        }
    }

    apply {
        if (meta.is_wireguard == 1 && hdr.wg_transport.msg_type == 4) {
            check_replay();
            if (meta.replay_detected == 1) {
                drop_packet();    // Dropped in NIC silicon — host never wakes up
            } else {
                send_to_host();   // Legit packet → DMA to host ring buffer
            }
        }
    }
}

1.2 — Nvidia BlueField-3 DPU: Full VPN Offload

The BlueField-3 is an ARM CPU + ConnectX-7 NIC on one PCIe card. You run a full Linux OS on the NIC. Your VPN daemon runs on the DPU's ARM cores, completely isolated from the host. Host compromise doesn't affect the VPN — the DPU has its own trust boundary.

Host Server (x86) BlueField-3 DPU ├── Application workloads ├── ARM Cortex-A78 (8 cores) ├── Guest VMs ├── Runs full Ubuntu Linux └── Connected via PCIe ├── WireGuard daemon here ├── Crypto keys NEVER on host ↕ PCIe ├── ConnectX-7 @ 200Gbps └── Physical NIC ports Traffic flow: External → NIC → DPU ARM (decrypt) → PCIe → Host Host → PCIe → DPU ARM (encrypt) → NIC → External Host root cannot access DPU memory → Zero-trust host model
BlueField-3 + WireGuard: ~100 Gbps encrypted throughput consuming 0 host CPU cycles. The host sees only plaintext traffic on a virtual NIC. Keys live in DPU memory, inaccessible to host OS — even if the host is fully compromised by a VM escape or kernel exploit.

02 FPGA Packet Processing — Custom Silicon at <1μs Latency HARDWARE

2.1 — Why FPGAs for VPN?

FPGAs (Xilinx/AMD Alveo, Intel Agilex) let you implement packet processing as actual digital circuits. No instruction fetch, no branch prediction, no cache misses — logic executes in parallel every clock cycle. A 300 MHz FPGA processes 300 million packet-decisions per second with deterministic sub-microsecond latency. Critical for trading firms, military, and ultra-low-latency VPN requirements.

// VHDL: ChaCha20 quarter-round — hardware implementation
// Executes in ONE clock cycle at 300MHz on Xilinx Alveo

VHDL
library IEEE;
use IEEE.STD_LOGIC_1164.ALL;
use IEEE.NUMERIC_STD.ALL;

entity chacha20_quarter_round is
    port(
        a_in, b_in, c_in, d_in : in  STD_LOGIC_VECTOR(31 downto 0);
        a_out, b_out, c_out, d_out : out STD_LOGIC_VECTOR(31 downto 0)
    );
end entity;

architecture rtl of chacha20_quarter_round is
    function rotl32(v : STD_LOGIC_VECTOR(31 downto 0); n : integer)
        return STD_LOGIC_VECTOR is
    begin
        return v(31-n downto 0) & v(31 downto 32-n);
    end;

    signal a1, b1, c1, d1 : STD_LOGIC_VECTOR(31 downto 0);
    signal a2, b2, c2, d2 : STD_LOGIC_VECTOR(31 downto 0);
    signal a3, b3, c3, d3 : STD_LOGIC_VECTOR(31 downto 0);
    signal a4, b4, c4, d4 : STD_LOGIC_VECTOR(31 downto 0);
begin
    -- All four ARX steps unrolled — pure combinational logic
    -- No clock dependency: output valid in single propagation delay (~3ns)
    a1 <= STD_LOGIC_VECTOR(unsigned(a_in) + unsigned(b_in));
    d1 <= rotl32(d_in XOR a1, 16);
    c1 <= STD_LOGIC_VECTOR(unsigned(c_in) + unsigned(d1));
    b1 <= rotl32(b_in XOR c1, 12);
    a2 <= STD_LOGIC_VECTOR(unsigned(a1) + unsigned(b1));
    d2 <= rotl32(d1 XOR a2, 8);
    c2 <= STD_LOGIC_VECTOR(unsigned(c1) + unsigned(d2));
    b2 <= rotl32(b1 XOR c2, 7);

    a_out <= a2; b_out <= b2; c_out <= c2; d_out <= d2;
end architecture;

-- Full ChaCha20 block = 20 quarter-rounds = instantiate 20 of these
-- Pipeline them → 1 block every clock cycle @ 300MHz
-- = 300M * 64 bytes = 19.2 Gbps from ONE FPGA resource block
